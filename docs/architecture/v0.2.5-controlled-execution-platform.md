# v0.2.5 Controlled Execution Platform

XFusion v0.2.5 introduces a manifest-first capability engine while preserving the
deterministic safety boundaries from v0.2.

## Execution Model

Capabilities are declared as YAML manifests and compiled into the same internal
`CapabilityDefinition` contract used by planning, policy, runtime validation,
redaction, audit, and verification.

The engine supports two execution bindings:

- `argv`: constrained command execution through a global executable allowlist.
- `python_adapter`: reviewed Python adapter execution for complex, privileged,
  dynamic, or multi-step behavior.

YAML never defines shell strings. `argv` bindings use only static tokens and
single typed argument tokens. There is no shell interpolation, command chaining,
glob expansion, redirect, or implicit `PATH` lookup.

## Mandatory Manifest Metadata

Every v0.2.5 manifest must include:

- input and output schemas
- risk tier and approval mode
- target constraints
- execution constraints
- side-effect classification
- redaction policy
- verification block

Manifest loading fails closed at startup for invalid schemas, duplicate
capability names, unknown executable IDs, unsafe argv tokens, missing
verification, or missing side-effect classification.

## Budget, Retrieval, And Explainability

`SessionExecutionBudget` provides session-level limits for steps, commands,
mutations, fallback attempts, exposed bytes, wall time, and cumulative risk.
Budget exhaustion produces a structured `budget_exceeded` audit result.

`CapabilityRetriever` exposes only a minimal capability surface to the agent. It
ranks candidates using semantic similarity, schema fit, success history, failure
penalty, risk penalty, side-effect penalty, platform availability, and recency.
Each candidate includes `capability_confidence` and a concise `why_selected`
explanation.

## Fallback Policy

Fallback execution remains constrained and auditable:

- template fallback is preferred
- restricted command fallback is argv-only and policy-gated
- agent-generated Python is disabled by default and requires an external sandbox
  before it can run

Fallback is not unrestricted shell access.

## Catalog Status

The repository includes a v0.2.5 catalog surface under `capabilities/v0.2.5` for
the supplied Linux operations capability set. These manifests are valid,
policy-visible capability declarations. Most catalog entries bind to
`python_adapter` IDs so their live behavior must be connected to reviewed
adapter implementations before production execution.

## Known Limitation Closure Plan

v0.2.5 is the controlled execution foundation. Manifest-complete does not mean
execution-complete. The remaining work is intentionally staged so new execution
surface lands only with tests and reviewed adapters.

Adapter execution closes in four tiers:

1. Tier 1: read-only diagnostics for system info, disk, memory, CPU, process
   lists, ports, interfaces, and routes.
2. Tier 2: service, log, and process inspection for systemd status/list,
   journald reads, bounded log search, and process inspection.
3. Tier 3: controlled mutation for service start/stop/restart, packages, users,
   and scoped file operations, all approval-bound.
4. Tier 4: high-risk admin operations for firewall, storage, backup restore,
   kernel modules, debugger attach, and traffic capture.

Unbound `python_adapter` IDs fail closed through `PythonAdapterRegistry` in
production mode. Catalog-inspection mode may return an explicit `unavailable`
result, but that result is not a successful execution and must not become a
mutation path.

Agent-generated Python stays disabled by default. Before enabling it, a real
sandbox must enforce CPU, memory, wall-time, filesystem, network, subprocess,
import, and output limits. The current `SandboxPolicy` is a static gate for
detecting forbidden file, network, subprocess, dynamic import, and import
allowlist violations before any runtime is attached.

Restricted fallback must reserve execution budget, include verification metadata,
use an allowlisted executable path, avoid shell semantics, pass policy review,
and produce an audit record. It is constrained recovery behavior, not a shadow
shell interface.

Capability retrieval should be fed from audited execution history. Ranking
penalizes unavailable adapters, higher risk, side effects, and prior failures,
and explainability records include nearby rejected alternatives.

Future TUI/debug work should expose the same state without overwhelming normal
mode: concise summary/result/interpretation in normal mode, and retrieval
candidates, confidence, ranking signals, policy, budget, fallback status,
verification, command trace, and audit ID in debug mode.
